# Linux Package Manager Distribution

- **Status:** Proposed
- **Date:** 2026-06-11
- **Issue:** [#324](https://github.com/wardian-app/Wardian/issues/324)
- **Decision owner:** Maintainers

## Context and Problem Statement

Wardian now has package-manager discovery on Windows through winget and on macOS
through the Wardian Homebrew tap. Linux still has only direct GitHub Release
installation snippets for the `.deb` and AppImage artifacts generated by the
stable release workflow.

Linux package-manager publishing has different trust and hosting choices than
winget and Homebrew. APT repositories, Flathub, and Snap all have real update
models, but they imply different security boundaries and maintainer work. Wardian
is a local command center that launches provider CLIs, watches user workspaces,
uses PTYs, and coordinates local files. That behavior maps most directly to a
traditional Debian package today and needs a deliberate sandbox design before
being published as Flatpak or Snap.

## Proposed Decision

Use a signed APT repository as Wardian's first Linux package-manager channel.
GitHub Releases remain the canonical release artifact source. The APT repository
is a package-manager mirror of the published stable `.deb`, not an independent
build system and not a replacement for the Tauri updater artifacts.

The intended public repository contract is `https://apt.wardian.org`. The
initial backend can be any static host that preserves that URL contract, such as
GitHub Pages behind the custom domain or object storage behind the same domain.
Do not publish user-facing APT install instructions until the domain, signing
key, fingerprint, and first validated repository tree are live.

The first public APT repository should publish:

- only stable releases;
- only the current `amd64` `.deb` while the release workflow emits only that
  Linux package architecture;
- a dedicated Wardian archive signing key;
- `InRelease` and `Release.gpg` signatures;
- a source-list snippet that uses per-repository `Signed-By` key scoping instead
  of global APT trust;
- validation commands that run against the generated repository metadata before
  upload.

The Debian package identity is part of the publishing contract. Before staging a
`.deb` into the repository, the publishing job must verify:

```bash
dpkg-deb -f Wardian_X.Y.Z_amd64.deb Package Version Architecture
```

Expected values:

- `Package`: `wardian`
- `Version`: `X.Y.Z`
- `Architecture`: `amd64`

Flathub should be deferred until Wardian can model its filesystem, PTY, provider
CLI, browser/WebKit, and external-editor permissions with a small permission
surface. Flathub requires stable software, required app metadata, a Flatpak
manifest, local builder/linter validation, and limited static permissions. Those
requirements are valuable, but they are not just a repackaging of the current
`.deb`.

Snap should be deferred. Wardian likely needs broad filesystem and process access
that would pressure the package toward classic confinement. Classic snaps require
store review and carry a higher security and support burden than a strict snap.
That work should only happen after a separate confinement design proves that a
strict snap is practical or that classic confinement is worth the review cost.

AppImage stays a direct-download artifact. Do not present AppImage as a
package-manager channel unless Wardian later adopts a real AppImage update
mechanism.

## Trust and Update Model

GitHub Releases remain authoritative for release version, asset names, download
URLs, SHA-256 digests, and Tauri updater metadata. The APT repository job must
consume the already-published stable `.deb` from GitHub Releases, verify that the
asset URL targets the requested tag, verify its SHA-256 digest, and only then
stage it into the repository layout.

APT trust is repository-level trust. The Wardian APT signing key signs repository
metadata, not every package file individually. Users trust the Wardian archive
maintainer key to preserve repository integrity, while the source release and
Tauri updater signatures remain separate release trust boundaries. The public
key fingerprint must be documented, and install instructions must bind the key to
the Wardian repository with `Signed-By`.

When Wardian is installed through APT, system package-manager updates own the
desktop app update path. The in-app updater may still report newer stable
versions, but it must not silently bypass APT ownership. Any future Linux in-app
installer behavior should detect package-manager installs before replacing files
outside the package database.

## Repository Shape

The generated repository should use a conventional minimal layout:

```text
apt/
  pool/main/w/wardian/Wardian_X.Y.Z_amd64.deb
  dists/stable/main/binary-amd64/Packages
  dists/stable/main/binary-amd64/Packages.gz
  dists/stable/Release
  dists/stable/InRelease
  dists/stable/Release.gpg
  wardian-archive-keyring.gpg
```

The first repository can use `stable` as both suite and codename. If Wardian later
adds preview/beta channels, use a separate suite instead of publishing prerelease
packages into `stable`.

Keep previously published `.deb` files in `pool/`; do not overwrite an already
published package version with different contents. If a package must be
republished for packaging-only reasons, publish a new Debian package version
instead of reusing the same version string.

## Release Automation Gate

Do not add upload/publish automation until the maintainers have provisioned:

- the APT repository hosting location;
- the public install URL, expected to be `https://apt.wardian.org`;
- the private archive signing key custody model;
- a rotation and recovery procedure for the archive signing key;
- a dry-run repository output directory that can be validated without publishing.

Local dry runs from Windows should use WSL Ubuntu or another Linux environment
for Debian tooling. The long-term publishing job should run on a Linux CI runner
so repository generation is reproducible and does not depend on a maintainer's
workstation.

The first automation step is a reusable GitHub Actions workflow that builds and
validates the APT repository from a published stable release. Manual runs should
default to `publish=false`. Stable release runs can call the workflow after
release publication with `publish=true`, but the workflow must skip effective
publication and upload a signed dry-run artifact when the real archive signing
key or host repository credentials are absent. The existing Wardian docs Pages
deployment must remain separate; if GitHub Pages hosts APT, use a separate
repository behind `https://apt.wardian.org`.

Once those exist, the release job can add a Linux-only post-publish step that:

1. reads `gh release view vX.Y.Z --json tagName,assets`;
2. verifies the `.deb` asset name, tag URL, and SHA-256 digest;
3. verifies package identity with `dpkg-deb -f`;
4. copies the `.deb` into `pool/main/w/wardian/`;
5. regenerates `Packages`, `Packages.gz`, and `Release`;
6. signs `Release` into both `InRelease` and `Release.gpg`;
7. validates metadata locally;
8. uploads the repository tree only after validation succeeds.

## Validation Commands

Repository generation should use Debian/Ubuntu tooling on a Linux runner:

```bash
npm run release:apt-repo -- \
  --release-assets dist/package-managers/vX.Y.Z/assets.json \
  --deb Wardian_X.Y.Z_amd64.deb \
  --out dist/apt \
  --signing-key "$WARDIAN_APT_SIGNING_KEY"
```

The script is a local/CI dry-run publisher. It validates the release manifest,
local `.deb` hash, Debian package name, Debian package version, and architecture
before staging the package. The underlying metadata commands are:

```bash
apt-ftparchive packages pool/main > dists/stable/main/binary-amd64/Packages
gzip -k -f dists/stable/main/binary-amd64/Packages
apt-ftparchive release dists/stable > dists/stable/Release
gpg --batch --yes --local-user "$WARDIAN_APT_SIGNING_KEY" --clearsign \
  -o dists/stable/InRelease dists/stable/Release
gpg --batch --yes --local-user "$WARDIAN_APT_SIGNING_KEY" -abs \
  -o dists/stable/Release.gpg dists/stable/Release
```

Validation before upload:

```bash
test -s dists/stable/main/binary-amd64/Packages
test -s dists/stable/main/binary-amd64/Packages.gz
gpg --verify dists/stable/Release.gpg dists/stable/Release
gpg --verify dists/stable/InRelease
```

End-to-end install validation should run in a disposable Debian or Ubuntu
container or VM. Once the public repository is live, use deb822 source syntax:

```bash
sudo install -d -m 0755 /etc/apt/keyrings
curl -fsSL https://apt.wardian.org/wardian-archive-keyring.gpg \
  | sudo tee /etc/apt/keyrings/wardian.gpg >/dev/null
sudo tee /etc/apt/sources.list.d/wardian.sources >/dev/null <<'EOF'
Types: deb
URIs: https://apt.wardian.org
Suites: stable
Components: main
Architectures: amd64
Signed-By: /etc/apt/keyrings/wardian.gpg
EOF
sudo apt update
apt-cache policy wardian
sudo apt install wardian
```

## References

- Debian `apt-secure(8)` documents signed repository metadata, `Signed-By`, and
  the repository maintainer trust boundary.
- Debian `apt-ftparchive(1)` documents generating `Packages` and `Release`
  indexes from `.deb` archives.
- Flathub submission and requirements docs document the manifest, metadata,
  stable-release, build, linter, and permission expectations.
- Snapcraft docs document channel release flow, strict confinement defaults, and
  the review/security implications of classic confinement.

## Consequences

- **Positive:** Linux package-manager work builds on Wardian's existing `.deb`
  release artifact instead of creating a separate Linux build pipeline.
- **Positive:** The trust model stays explicit: GitHub Releases are canonical,
  while APT metadata provides Linux package-manager integrity.
- **Positive:** Flathub and Snap are not blocked forever, but their sandbox
  decisions are not hidden inside a packaging PR.
- **Negative:** Linux package-manager publishing still needs hosting and signing
  key infrastructure before it can be automated.
- **Negative:** The first Linux package-manager channel primarily serves
  Debian/Ubuntu-family users; Flatpak/Snap discovery remains future work.
